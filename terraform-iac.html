Immutable Infrastructure (IaC)

Enterprise-scale Azure Hub & Spoke deployments using Terraform.

Architecture Overview

The Data Linq utilizes a Modular Framework to manage global Azure footprints. By separating core networking (Hub) from environment-specific workloads (Spokes), we eliminate configuration drift and ensure 1:1 parity between development and production.

Repository Structure

modules/
 ├── hub/ # Core Networking
 ├── peering/ # Connection Logic
 └── spoke/ # Workload LZ
stacks/
 ├── hub/
 ├── spoke-dev/
 ├── spoke-uat/
 └── spoke-prd/

The Methodology

  • State Isolation Each stack (Dev, UAT, Prod) manages its own backend state file, ensuring a failure in development can never impact production.
  • Modular Reuse Infrastructure patterns are defined once in modules/. Stacks then call these modules with environment-specific .tfvars.
  • Automated Handshakes The peering module automatically creates the bi-directional links required to attach new Spokes to the central Hub.

Environment Anatomy: spoke-dev

A closer look at the file-level structure within a single environment stack. This separation of concerns allows for clean audits and scalable deployments.

stacks/spoke-dev/
├── backend.hcl
├── providers.tf
├── main.tf
├── storage.tf
├── terraform.tfvars
├── .terraform.lock.hcl
└── tfplan (generated)

File Breakdown

  • backend.hcl: Defines the remote state location (Azure Storage) to enable team collaboration and state locking.
  • main.tf: Acts as the orchestrator, calling the standardized modules for networking and security.
  • storage.tf: Dedicated resource file for ADLS Gen2, Blob containers, and lifecycle management policies.
  • terraform.tfvars: Injects environment-specific variables such as IP ranges, region, and resource naming conventions.
