VNet Injection: Taking Control

By default, Databricks manages the virtual network. In an enterprise environment, we use VNet Injection to deploy Databricks clusters into a pre-configured VNet managed by the customer. This allows us to apply custom Network Security Groups (NSGs) and route traffic through a central firewall.

The Private Link Advantage

Standard data access often goes over the public internet via service endpoints. Our pattern utilizes Azure Private Link to provide private IP addresses for:

  • Back-end Connectivity: Private access to ADLS Gen2, Key Vault, and SQL Databases.
  • Front-end Connectivity: Ensuring users can only access the Databricks UI from within the corporate VPN or ExpressRoute.
  • Web Auth: Secure authentication handshakes that never touch public DNS.

Infrastructure Requirements

// Required Subnets for Databricks VNet Injection
- public-subnet: For cluster management traffic (/26 or /24)
- private-subnet: For worker-to-worker communication (/26 or /24)
- privatelink-subnet: For hosting Storage & KeyVault Endpoints

Security Benefits

  • Data Exfiltration Protection: By disabling public access on Storage Accounts and using Private Endpoints, we ensure data cannot be moved to unauthorized accounts.
  • Custom Routing (UDR): Forced tunneling of all egress traffic to an Azure Firewall for Deep Packet Inspection (DPI).
  • Zero Public IPs: Clusters can be deployed with No Public IP (NPIP) enabled, removing the attack surface entirely.
